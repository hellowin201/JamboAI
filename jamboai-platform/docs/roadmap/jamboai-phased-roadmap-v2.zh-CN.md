# JamboAI 分阶段开发路线图 V2

## 规划规则

本路线图替代之前线性的六阶段拆分。新版保留阶段 0，然后围绕可交付业务切片组织后续工作，方便 Codex 分阶段实现和验证。

RuoYi-Vue-Plus 已有功能应尽量复用。除非 JamboAI 需要扩展点，不要重写系统用户、角色、菜单、字典、OSS、日志、监控或代码生成功能。

## 阶段 0：工程基线

目标：让代码库具备稳定的模块化开发基础。

范围：

- 使用最新 83 表 SQL 作为唯一 schema 基线。
- 所有 JamboAI 业务代码放在 `org.dromara.jamboai.*`。
- Maven 模块名、前端项目名和文档命名统一使用 `jamboai`。
- 确认 API 根前缀：`/api/sys/**`、`/api/mch/**`、`/api/usr/**`、`/api/pub/**`、`/api/whk/**`。
- 建立业务上下文：`tenant_id`、`agent_id`、`merchant_id`、`staff_id`、`member_id`、`channel_type`。
- 建立 WhatsApp、支付和模型供应商适配器接口。
- 建立 mock/sandbox/production 环境切换规则。
- 初始化 10 种支持语言。
- 保持后端、后台前端、商户端和用户端构建验证。

验收：

- 后端构建成功。
- 后台前端构建成功。
- 商户端构建成功。
- 用户端构建成功。
- `infra/sql` 中 schema 表数量为 83。
- `docs` 下存在架构和 API 文档。

## 阶段 1：基础、组织、权限和国际化

目标：让平台管理和商户身份体系可用。

后端范围：

- 国家租户扩展。
- 城市代理。
- 商户。
- 商户员工。
- 商户角色和权限。
- 会员基础资料。
- 商户会员关系。
- 字典、国家、城市、币种、时区、行业和语言数据。
- 国际化消息框架。
- AI、渠道、支付和文件的平台配置框架。

前端范围：

- `admin-ui`：租户扩展、城市代理、商户、员工、会员和字典页面。
- `merchant-mobile`：商户员工登录、个人资料、按角色显示的壳和首页。
- `customer-mobile`：在共享认证基础设施需要时，只做基础用户登录/注册壳。

RuoYi 复用：

- 复用 `sys_user`、`sys_role`、`sys_menu`、`sys_dept`、`sys_tenant`、`sys_dict`、`sys_config`、`sys_oss`、日志和权限。
- 扩展 RuoYi 租户和菜单逻辑，不重写框架。

验收：

- 平台后台可以创建租户扩展、城市代理和商户。
- 商户员工可以独立于 `sys_user` 登录。
- 商户员工数据按商户隔离。
- 后台菜单和核心 App 菜单支持 10 种语言标签。

## 阶段 2：商户优先的通信 MVP

目标：让商户可以接收、查看和回复会话。

后端范围：

- 渠道账号。
- WhatsApp 号码。
- WhatsApp mock/sandbox/production 配置。
- WhatsApp Webhook 接收和校验。
- 会话。
- 消息。
- 人工接管。
- 消息模板。
- 入站消息 Spring Event 处理。

前端范围：

- `admin-ui`：渠道账号、WhatsApp 号码、Webhook 日志和消息日志页面。
- `merchant-mobile`：会话列表、消息视图、未读状态、人工回复和接管提醒。
- `customer-mobile`：只在需要时保留基础客户入口壳。

验收：

- mock WhatsApp 入站消息可以创建或更新会话。
- 商户端可以看到会话并回复。
- AI/机器人模式转人工模式时，商户员工可见提醒。
- Webhook 处理具备幂等性。

## 阶段 3：商品、服务和订单 MVP

目标：让会话连接真实商业数据。

后端范围：

- 商品。
- SKU。
- 服务。
- 服务排期基础能力。
- 订单。
- 订单明细。
- 订单流转。
- 必要的预约和核销基础能力。
- 仅提供 mock 支付意图占位。

前端范围：

- `admin-ui`：商品、服务和订单监管页面。
- `merchant-mobile`：商品管理、服务管理和订单管理。
- `customer-mobile`：商品浏览、下单和订单详情。

AI 准备：

- 创建未来 AI 工具可调用的业务服务接口：
  - 搜索商品。
  - 推荐商品。
  - 创建订单草稿。
  - 查询订单。
  - 查询服务排期。

验收：

- 商户可以创建商品或服务。
- 用户 App/H5 可以浏览并创建订单。
- 商户可以管理订单。
- WhatsApp 会话可以关联订单草稿上下文。

## 阶段 4：AI Agent 和知识库 MVP

目标：让 AI 真正服务客服和交易引导。

后端范围：

- 模型供应商。
- 模型路由。
- 提示词模板。
- 模型调用日志。
- Token 日志。
- Agent 能力。
- Agent 工具。
- Agent 模板。
- 商户 Agent 应用。
- Agent 任务。
- 工具调用日志。
- 知识文档、切片、向量和 FAQ。

前端范围：

- `admin-ui`：模型供应商、路由、提示词、Agent 模板、工具和调用日志。
- `merchant-mobile`：AI 开关、Agent 应用配置、FAQ/知识库基础能力和 AI 接管状态。
- `customer-mobile`：在适用场景中支持 AI 辅助客户会话。

工具规则：

- AI 必须调用受控工具，不能直接调用 repository 或 mapper。
- 工具调用必须校验租户、商户、渠道和用户范围。
- 高风险动作必须要求确认。

验收：

- 商户可以启用已配置的 AI Agent。
- AI 可以基于 FAQ/知识库回答。
- AI 可以调用工具搜索商品并创建订单草稿。
- 低置信度或不支持的请求可以转人工。

## 阶段 5：支付和交易闭环

目标：让订单可以通过可配置支付服务商完成支付。

后端范围：

- 支付渠道。
- 平台账号。
- 商户支付账号。
- 支付交易。
- 支付回调日志。
- 支持 mock/sandbox/production 模式的 Flutterwave 适配器。
- 支付链接创建。
- 订单支付状态事件处理。
- 基础风控校验。

前端范围：

- `admin-ui`：支付渠道、平台账号、回调日志和交易页面。
- `merchant-mobile`：商户支付账号、支付状态和订单支付视图。
- `customer-mobile`：支付链接入口和支付结果页。

验收：

- mock 支付可以完成订单端到端闭环。
- Flutterwave sandbox/production 配置可以无代码修改切换。
- 重复回调不会重复记支付成功。
- 订单支付状态通过支付领域事件更新。

## 阶段 6：财务、风控和统一数据智能

目标：从 MVP 进入可运营商业平台。

后端范围：

- 钱包。
- 钱包账本。
- 费用规则。
- 费用账本。
- 分佣账本。
- 结算。
- 提现。
- 对账单和对账明细。
- 风险评分。
- 黑名单。
- UDI 记忆、反馈、指标和行为。

前端范围：

- `admin-ui`：结算、提现、对账、风控、行为和指标看板。
- `merchant-mobile`：余额、结算、提现和经营指标。
- `customer-mobile`：反馈和基于行为的个人推荐能力。

执行说明：

阶段 6 较大，实际开发建议拆为：

- 阶段 6A：钱包、账本、费用、分佣、结算、提现和对账。
- 阶段 6B：风控、记忆、反馈、指标、行为和分析看板。

验收：

- 资金流动有账本记录。
- 平台可以计算费用和分佣。
- 商户可以查看结算和提现状态。
- 运营人员可以看到转化率、AI 命中率、转人工率、支付率和订单指标。
